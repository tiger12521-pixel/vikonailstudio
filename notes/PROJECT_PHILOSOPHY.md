# 專案維護理念

## 1. 先保持架構，再增加功能

功能能執行不代表適合直接加入。新增內容前，先決定它應放在哪個 HTML、CSS、JavaScript 或 API 模組。

## 2. 小步修改、逐階段驗證

每次只處理一個明確目標：

1. 修改。
2. 本機測試。
3. 查看 Git Diff。
4. Commit。
5. Push。
6. 驗證線上版本。

這樣出現 Bug 時能快速定位是哪一階段引入。

## 3. 不依賴記憶

重要操作、決策與發版流程必須寫入文件。即使幾年後回來，也不應靠回想聊天內容才能維護。

## 4. 不讓入口檔案持續膨脹

- `css/style.css` 只負責載入順序。
- `js/main.js` 只負責初始化。
- 新功能應新增獨立模組，而不是方便地塞進入口檔案。

## 5. 畫面穩定優先於重構漂亮

任何重構都必須以「功能與顯示不變」為第一優先。若無法確定等效，應拆成更小步驟測試。

## 6. 正式版本必須可辨識與可回復

每個正式版本需要：

- 清楚的版本號。
- Console 版本資訊。
- Git Commit。
- Git Tag。
- 更新紀錄。

## 7. 文件也是產品的一部分

若程式可以執行，但沒有人知道如何啟動、發布或修改，它仍不是完整且可長期維護的專案。
